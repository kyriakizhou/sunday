730+ Java Application Instances Performance Tuning

YourKit / flame graph / thread count / JVM args / redundant network calls

I worked on performance tuning for a job scheduler application running across 730+ Java instances. The application was seeing P95 latency around 1.5K ms, which was slow enough to create a large work backlog and delay important subscription jobs such as monthly auto-renewal processing.

Goal

Before blindly increasing compute, I was asked to find whether we could improve latency and throughput from the application side. The team had little prior performance-tuning context for this service, so the first task was to understand where time and resources were actually going. I set an initial target of reducing latency by 10%, matching the P95 improvement benchmark from a related application in the organization.

Profiling

I used YourKit to inspect live Java process behavior and flame graphs to visualize where CPU time and waiting time were concentrated. A flame graph made the system much easier to reason about: the vertical axis showed the call stack from the root entrypoint up to the Java methods doing work, while the horizontal width showed how frequently a stack appeared in samples.

That view helped me understand the app quickly because it exposed the dominant stack traces directly. It also revealed repeated network and database fetches inside the same call path, including multiple fetches with the exact same parameters.

Changes

  • Standardized worker threads. Half the fleet was running with 32 worker threads while the other datacenter ran with 16. YourKit showed many threads blocked or waiting, so I moved the 32-thread instances down to 16 threads. Latency dropped by 22% immediately and throughput became more consistent across datacenters.
  • Validated the lower bound. I tested reducing both sides to 8 worker threads. It gave a small additional latency boost, but throughput dropped, so I kept the fleet at 16 threads.
  • Investigated blocked threads. A deeper inspection showed the application spent significant time blocked around a heavily used content-library fetch path, including a volatile field in the dependency's fetch library.
  • Tuned JVM flags. I set -XX:MaxMetaspaceSize=640m, -XX:+UseG1GC, -XX:+UseNUMA, and -XX:UseAVX=2 to cap class metadata growth, reduce long garbage collection pauses, align better with NUMA hardware, and use vectorized CPU instructions where available.
  • Reduced logging overhead. I changed log retention behavior to avoid unnecessary zipping and unzipping, reduced the size of each log file kept on disk, and reduced the number of retained log files.
  • Tracked redundant data fetches. Flame graphs surfaced duplicate database calls in the same trace stack, giving us concrete cleanup targets instead of vague suspicions about network overhead.

Result

The tuning work reduced P95 latency from roughly 1.5K ms to 1K ms, an approximately 27-30% improvement, and increased throughput by 5%. The biggest single win came from right-sizing worker threads, but the larger impact came from making the system observable enough to tune deliberately instead of guessing where the bottlenecks were.
